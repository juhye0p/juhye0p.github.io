Embedded System Kernel Recovery
2026-04-01 04:53:46 #IoT

Introduction

This article discusses the restoration of corrupted embedded kernel using an SD card (Secure Digital card).

How I destroyed the kernel image

In my case, I attached an 8-pin SOIC clip directly to the NOR flash chip while the SoC was still on the board, which caused bus contention between the SoC and the SPI programmer. This likely resulted in unintended writes to the NOR flash, corrupting the kernel image and causing the boot sequence to fail during the checksum verification stage.
Using an 8-pin SOIC clip to restore the kernel image is not impossible, but I was just scared to use it again. So I found another way to restore the kernel image using an SD card.

To restore the corrupted kernel image, I should extract the kernel image from the original firmware and flash it into the kernel partition of the NOR flash memory. I used the original firmware, that I extracted before.
By the way, In case you can only download a legacy firmware online, I believe flashing an entire legacy firmware in NOR flash chip and updating it to a stable firmware using apps, after reviving the device, is the one of the firmware restoring method.

On the first attempt, I extracted the kernel image through binwalk from the original firmware and flashed it into the kernel partition of the NOR flash memory. However, the device still failed to boot it.
This is because binwalk extracts only the payload data (1,568,528 bytes) without the 64-byte uImage header. The uImage header contains critical metadata including the CRC checksum, load address, and entry point — without it, U-Boot cannot recognize the data as a valid kernel image.
Therefore, I had to extract 1,568,592 bytes (0x1E86A7 - 0x69757 = uImage header + kernel data) from the kernel image offset to include the complete uImage structure.

% dd if=[firmware] of=kernel.bin skip=431959 count=1568592 bs=1

After extracting the kernel image, I had to flash it into the kernel partition.
By checking the mtdparts environment variable through the bootloader shell using printenv command, I could see that the linux partition, which contains the kernel image, starts at address 0x100000 and has a size of 0x1b0000.

Therefore, I had to flash the kernel image extracted earlier into the linux partition. In my case, I used the fatload command to load the kernel image into the device RAM, so I had to format the SD card as FAT32 in advance.

After formatting the SD card, I copied the kernel image I extracted earlier to the SD card.

% cp ./kernel.bin /Volumes/SDCARD/

Restoring the kernel image

Back to the bootloader shell, first I attached the SD card to the device, and then I checked if the device could recognize the SD card and set the current MMC (MultiMediaCard) device through the mmc command. In my case, the MMC index of the SD card was 0.

nvt@: mmc dev 0
mmc0 is current device

After setting the current MMC device, I loaded the kernel image, which I stored in the SD card, into 0x00000000 address of the device RAM through the fatload command.

nvt@: fatload mmc 0:1 0x00000000 kernel.bin
1568592 bytes read in 77 ms (19.4 MiB/s)

On the second attempt, I tried to boot the device from 0x00000000 address of RAM through bootm command to see the result of the checksum verification. The checksum verification of the kernel image succeeded. However, as expected, the boot sequence still failed.

nvt@: bootm 0x00000000
## Booting kernel from Legacy Image at 00000000 ...
   Image Name:   Linux-4.19.91
   Image Type:   ARM Linux Kernel Image (uncompressed)
   Data Size:    1568528 Bytes = 1.5 MiB
   Load Address: 00008000
   Entry Point:  00008000
   Verifying Checksum ... OK
ERROR: can't get kernel image!
nvt@:

The checksum was valid, but bootm still failed because the kernel image at 0x00000000 overlaps with the load address 0x00008000 — the image spans 0x00000000–0x17EF90, and U-Boot needs to relocate it to 0x00008000, which falls within the source range. This overlap prevents U-Boot from safely copying the kernel to its load address.
To flash the NOR flash memory, I used the sf command and set the current SPI (NOR) flash chip. In my case, the index of the NOR flash chip was 0.

nvt@: sf probe 0
SF: Detected GigaDevice 25Q128E ...(omitted)

After setting the current NOR flash chip, I flashed kernel image, which is loaded on 0x00000000 address of the device RAM, into 0x100000 address of NOR flash memory, which is kernel partition, 0x1b0000 bytes. 

nvt@: sf update 0x00000000 0x100000 0x1b0000
device 0 offset 0x100000, size 0x1b0000
nor0 0x10000 0x100

Updating, 89% 2403899 B/snor0 0x10000 0x100

Updating, 97% 1584768 B/snor0 0x10000 0x100

After flashing kernel partition, finally I could boot the device through nvt_boot command. I could see the bootcmd environment variable through printenv command.
Additionally, you can also flash the entire firmware into the NOR flash memory through the same method. If the bootloader is also corrupted and cannot load anything, perform a chip-off extraction (optional) and use an 8-pin SOIC clip to flash the entire firmware into the NOR flash chip.
